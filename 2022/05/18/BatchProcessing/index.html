批次處理

發表於 2022-05-18 | 分類於 Database

批次處理系統

批次處理（Batch Processing）的作業方式，是先蒐集資料，當資
料累積到一個程度時，再一次處理所有的資料。批次處理型的作業系
統，適合處理週期性的大筆資料。

UNIX

UNIX是一套作業系統，就如同 Windows XP、Windows Server 2003 或 Mac OS 一樣，負責管理電腦所有資源，包括處理器、記憶體、週邊設備、檔案等。

使用者透過應用程式或作業系統來操控電腦硬體，作業系統再將執行的結果回應給使用者或應用程式，因此可以說，作業系統是電腦硬體與使用者/應用程式之間的溝通橋樑。

Unix哲學

  • 每個程式一次只做好一件事。 最佳代表：Sort
  • 預期每個程式的輸出都將會變成另一個未知程式的輸入。不要讓無關資訊干擾輸出。
  • 設計和建構軟體時儘早嘗試，且重建時不要猶豫
  • 優先使用工具來幫助減輕程式設計任務的負擔。

Unix 最大的侷限：單台機器上執行，因此給Hadoop一個切入點，一個能夠儲存並管理大量資料的平台。

MapReduce

簡介

  • 跟Unix工具有幾分相似，但是卻可以分佈在數千台機器上。
  • 一個MapReduce相當於一個Unix程序，接受一個或多個輸入，然後產生一個或多個輸出。
  • 由Google提出，用來處理大量資料的軟體設計模型，批次處理演算法。
  • MapReduce作業是在分散式檔案系統上讀寫作業

分散式檔案系統HDFS(Hadoop Distributed File System)

  • Hadoop實作的 MapReduce檔案系統被稱HDFS(Hadoop Distributed File System)，重新實作ＧＯＯＧＬＥ檔案系統ＧＦＳ的開放原始碼實作版本。最初由Yahoo提出
    • 存儲大數據
    • 為應用提供大數據高速讀取的能力
  • 主要特點是通過將數據和計算分布在集群中的各節點伺服器，達到分布式計算的目的。
  • 計算邏輯和需要用到的數據在就近的節點，分區並行計算後再匯總(Map & Reduce)。

HDFS架構

和HDFS 打交道是通過一個client library. 無論讀取一個文件或者寫一個文件，我們都是把數據交給HDFS client，它負責和Name nodes 以及Data nodes 聯繫並傳輸數據

  • Name Nodes：類似中央伺服器，負責跟蹤哪些檔案區塊( data block)儲存在哪台機器上
  • Data Nodes：存儲data block 的機器，在讀寫過程中，Data nodes 負責直接把用戶讀取的文件block 傳給client，也負責直接接收用戶寫的文件。

當我們讀取一個文件時：

  1. Client聯繫 Name nodes，得到文件的 data block组成、以及每個 data block 所在的機器以及具體存放位置
  2. Client 聯繫 Data nodes，進行讀寫操作

MapReduce流程


💡 MapReduce其實是一種開發模式（Program Model），基本上可以把整個邏輯分成為Map階段和Reduce階段。

  • Map：基本上會做filtering和sorting並且傳出一個key value pair做結果

（以word count為例，每一個字會作為最後的key，而value則是1代表有一筆）

  • Reduce：基本上會做整合

（以word count為例，從Map傳過來的key如果一樣，表示同一個字，因此把一樣的key做加總最後的出總筆數）

MapReduce的Joins

然而，MapReduce並沒有索引的概念！當給MapReduce作業一組檔案輸入時，他會讀取這些檔案全部的內容，資料庫將此操作稱為“全表掃描”，如果資料量很大還全表掃描其實挺可怕！

Hadoop支持兩種Join，以基於某個列連接兩個或多個數據集。Map-side join和Reduce-side join。

  • Map-side join：當一個數據集很大而另一個數據集很小時使用
  • Reduce-side join：可以連接兩個大數據集。

Map-side join

將數據從不同的dataset中取出，連接起來並放到相應的某個Mapper中處理，因此key相同的數據肯定會在同一個Mapper裡面一起得到處理的。

Reduce-side join

Map端的產出結果分兩類：有哪些Key，以及每個Key有哪些值。Reducer收到後，自己依Key去巡覽各Key的明細產生關聯結果
